SECTION E — ALGORITHMIQUE : PATHFINDING (router.py)  *** LE CŒUR ***
--------------------------------------------------------------------------------

[ ] E.1  DIJKSTRA : principe général. Pourquoi Dijkstra et pas un simple
         BFS (parce que les coûts diffèrent : restricted = 2). Différence
         BFS / Dijkstra / A*.
[ ] E.2  heapq : pourquoi c'est autorisé (file de priorité = structure de
         données générique, PAS une lib de graphes). Ce qu'est un tas
         (heap) binaire, pourquoi heappop renvoie le plus petit.
[ ] E.3  La clé du tas = tuple (cost, penalty, name). Expliquer CHAQUE
         composante :
           - cost    : coût total en tours
           - penalty : +1 par zone NON priority -> départage en faveur
                       des zones priority à coût égal (exigence VII.1)
           - name    : pour rendre l'ordre déterministe
[ ] E.4  "Stale entry" : pourquoi on teste `if (cost, penalty) > best[name]:
         continue`. Notion d'entrée périmée dans le tas (lazy deletion).
[ ] E.5  _build_path : reconstruction du chemin via le dict predecessors
         (on remonte de end vers start puis on inverse).
[ ] E.6  Path.cost : pourquoi on somme movement_cost de toutes les zones
         SAUF la première (le start est le point de spawn, pas "entré").
[ ] E.7  BLOCKED : pourquoi une zone bloquée n'est jamais explorée
         (is_accessible()), donc aucun chemin valide ne la traverse.
[ ] E.8  CHEMINS MULTIPLES — find_disjoint_paths : stratégie gloutonne à
         capacité résiduelle (inspirée flow-based / lem-in). On prend le
         plus court, on "consomme" 1 unité de capacité sur chaque zone et
         connexion traversée, on recommence. Une zone/connexion sort du jeu
         quand elle est saturée.
[ ] E.9  Pourquoi un même chemin peut être renvoyé plusieurs fois (= une
         "lane" par unité de capacité parallèle).
[ ] E.10 path_throughput : débit d'un chemin = min des capacités le long
         du chemin (goulot d'étranglement).
[ ] E.11 DISTRIBUTION — PathPlanner.assign_drones_to_paths : règle
         "earliest-arrival". Une lane portant déjà k drones livre le
         suivant au tour k + lane.cost. On place chaque drone sur la lane
         d'arrivée la plus précoce. Pourquoi ça minimise le nombre de tours.
[ ] E.12 NoSolutionError : quand est-elle levée (des drones à router mais
         aucune lane disponible).
[ ] E.13 COMPLEXITÉ : un Dijkstra = O(E log V). Les chemins sont calculés
         UNE FOIS au départ (pas recalculés à chaque tour). Savoir
         répondre à "et avec 1000 drones ?".

SECTION F — SIMULATION TOUR PAR TOUR (simulator.py)  *** LE CŒUR ***
--------------------------------------------------------------------------------

[ ] F.1  Boucle run() : tant que tous les drones ne sont pas arrivés, jouer

[ ] F.2  LIBÉRATION SIMULTANÉE (règle VII.3) : pourquoi on traite les drones

         même tour (la place libérée est vue tout de suite par can_accept).
[ ] F.3  TRANSIT RESTRICTED (2 tours) : un drone occupe la connexion au
         tour T (affiché D<ID>-<connexion>), puis arrive dans la zone au
         tour T+1 (D<ID>-<zone>). Il ne PEUT PAS attendre sur le lien.
[ ] F.4  Le dict `incoming` : pourquoi il faut réserver à l'avance une place
         dans la zone restricted pour le tour T+1 (sinon on surbooke). On
         ne s'engage sur le lien que si occupancy + incoming < max_drones.
[ ] F.5  Drone qui attend : il reste sur place et est OMIS de la ligne du
         tour (exigence VII.5).
[ ] F.6  Détection de deadlock : max_turns dépassé -> RuntimeError au lieu
         d'une boucle infinie.
[ ] F.7  snapshots vs history : history = les mouvements affichés ; snapshots
         = position de TOUS les drones à chaque tour (utilisé par le
         renderer graphique).
[ ] F.8  (si ajouté) stats() : turns, coût total, tours moyens/drone,
